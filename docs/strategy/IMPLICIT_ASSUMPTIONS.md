# Scaling Intelligence — Implicit Assumptions

**Purpose:** OPEN_QUESTIONS_LOG.md and ARCHITECTURAL_REVIEW.md both catalog things that were *raised but not yet resolved*. This document is different in kind: it catalogs things that were never raised at all, because they were operating as unstated defaults from the start. Each one is here because a new collaborator reading FOUNDATION.md and the four Knowledge Cards, without having watched the work happen, would plausibly draw a different — and wrong — conclusion.

For each: the assumption as it has actually been operating, why a newcomer would likely misread it, and what should be stated explicitly instead.

---

### 1. "Innovation scaling" has quietly meant "CGIAR agricultural innovation scaling"

**How it's actually been operating:** Every document ingested, every case discussed, every capability gap identified is CGIAR or CGIAR-adjacent. The four AI roles (Analyst, Coach, Portfolio Intelligence) were derived entirely from CGIAR-specific mechanics — PRMS, stage-gating, IR/IU scoring.

**What a newcomer would misread:** FOUNDATION.md's vision and purpose sections are written in domain-general language ("innovation scaling," "a project team, a coach, a funder, or a portfolio manager") with no CGIAR qualifier anywhere in Sections 1–2. A new collaborator would reasonably assume Scaling Intelligence is meant to work across health, education, climate, or any other scaling domain today, and would be surprised to discover the entire evidence base — and even the shape of the Capability Framework — is agricultural-development-specific and has never been tested against a single non-CGIAR source.

**What should be stated explicitly:** FOUNDATION.md should say plainly that the current repository is scoped entirely to CGIAR's agricultural innovation ecosystem, that domain-general applicability is an aspiration, not a demonstrated property, and that methodology neutrality has so far only been tested *within* a fairly narrow, related family of traditions — not across genuinely different domains.

---

### 2. "The repository" and "the source skill library" are different things — but neither has a real, persistent home yet

**How it's actually been operating:** The Librarian reads source documents from a pre-existing connected library it does not own or modify. Everything the Librarian produces — every Knowledge Card, the Reflection, FOUNDATION.md itself — lives as standalone files in a session output folder. Nothing has been committed to version control, a database, or any system designed for long-term persistence.

**What a newcomer would misread:** FOUNDATION.md talks about "the repository" as though it is a stable, findable thing with a location. A new collaborator would likely assume there is a GitHub repo, a folder, or a database somewhere that constitutes "the repository," and could easily spend real time looking for it before realizing that, as of this Foundation Edition, the repository's actual persistence mechanism has never been decided.

**What should be stated explicitly:** Where the repository is meant to actually live going forward — GitHub, as this document's intended destination implies, or something else — and who is responsible for assembling the loose files produced so far into that real location.

---

### 3. Cross-card knowledge has been carried in conversation memory, not verified against the saved files themselves

**How it's actually been operating:** Every claim in every card about "previously curated cards" — relationships, confidence upgrades, cross-references — was generated by recalling earlier turns in this same conversation, not by re-opening and re-reading the saved Knowledge Card files from disk.

**What a newcomer would misread:** This is the single most consequential item on this list. A new collaborator — or a fresh AI session with no memory of this conversation — would only have the saved files to work from. If any claim about "what the repository already knows" exists only in this conversation's turns and was never fully written into a saved file, that knowledge is effectively gone the moment this session ends. Nothing has been done to confirm the saved files are fully self-sufficient without conversational memory behind them.

**What should be stated explicitly:** A standing rule that the repository must be able to be picked up cold, from files alone, with no dependency on any particular conversation's memory — and, practically, a one-time check of whether the current four cards plus FOUNDATION.md actually satisfy that, or whether some nuance only exists in chat responses that were never copied into a file.

---

### 4. One source document has always produced exactly one Knowledge Card

**How it's actually been operating:** Every ingestion so far has been a strict 1:1 mapping — one uploaded file, one card. This was never decided as a rule; it simply happened four times in a row.

**What a newcomer would misread:** A new collaborator handed a source that bundles several distinct things — a slide deck containing multiple independent case studies, for instance — would have no guidance on whether to split it into multiple cards, merge it with a related companion document into one card, or follow the default and card it as a single unit regardless of internal heterogeneity.

**What should be stated explicitly:** Either confirm 1:1 as the standing default and give explicit conditions for departing from it, or acknowledge it was never actually a deliberate choice and should be decided now.

---

### 5. The Librarian has only ever been reactive — it has no ability to independently find or fetch new sources

**How it's actually been operating:** All four documents were ingested because they were uploaded. FOUNDATION.md's "highest-priority next steps" name specific documents to acquire next, phrased with agency ("Ingest the source believed to underlie...").

**What a newcomer would misread:** That phrasing implies the Librarian will go get those documents. In practice, the Librarian has never searched for, requested, or retrieved anything on its own — every acquisition has required a human to locate the file and hand it over. A new collaborator could reasonably expect the "priority next steps" to execute themselves, or could be confused about why a clearly-identified, already-located file (like the one already sitting in the connected skill library) hasn't simply been ingested yet.

**What should be stated explicitly:** The Librarian's acquisition role is currently entirely passive; "priority next steps" are a request list for a human, not a task queue the system executes on its own.

---

### 6. Someone has to have the authority to resolve what the Librarian flags as open questions — and it's never been said who

**How it's actually been operating:** Several genuinely unresolved questions have been logged (the AoW2/S4I relationship, taxonomy reclassification) with an implicit expectation that a human will eventually resolve them. In practice, that has meant whoever is directing the Librarian's work in the moment.

**What a newcomer would misread:** A new collaborator wouldn't know whether they personally have standing to resolve a flagged open question, approve a recommended Refresh, or change the taxonomy — or whether those decisions require someone else's sign-off, given this is describing CGIAR institutional knowledge that a single collaborator may not have full authority over.

**What should be stated explicitly:** Who is actually authorized to resolve repository-level open questions and approve Refresh actions, even if the honest current answer is simply "whoever is directing the work at the time."

---

### 7. "Foundation Edition" has no defined exit criteria

**How it's actually been operating:** The project has been described as being in a "Foundation Edition" stage throughout, treated as self-evidently early — small number of documents, core discipline established but the multi-role platform unbuilt.

**What a newcomer would misread:** Without stated criteria, a new collaborator has no way to know what would need to become true for the project to no longer be "Foundation Edition" — a certain number of cards, the Analyst role existing, a first cross-methodology validation completed — and could either declare graduation prematurely or never recognize that it's already happened.

**What should be stated explicitly:** Concrete, checkable criteria for what "beyond Foundation Edition" looks like.

---

### 8. Confidence grades reflect one curator's judgment, not a calibrated or independently checked standard

**How it's actually been operating:** Every "High," "Medium," or "Low" confidence label, and every Observed/Interpreted/Hypothesized distinction, has been assigned by a single curator's read of the material, with no second reviewer and no defined criteria distinguishing, say, Medium from Medium-high beyond that curator's own sense of it.

**What a newcomer would misread:** A new collaborator would likely treat these labels as more objective and reproducible than they actually are — assuming, reasonably, that "Medium confidence" means something consistent across cards, when it currently just means "what one reader judged, once, with no calibration check against anyone else's independent read of the same source."

**What should be stated explicitly:** That confidence grading is currently single-curator judgment, not a calibrated standard, and — if reproducibility matters — what a calibration process would need to look like.

---

### 9. The curation process has assumed source material is extractable text, in English

**How it's actually been operating:** Every source has been a PDF or DOCX processed by extracting its text. Several cards separately flagged, as one-off issues, that a specific slide or diagram carried meaning the text extraction missed — but this was treated each time as a local quirk of that one document, not stated as a general, standing limitation of the method itself. No source in a language other than English, and no primarily visual, audio, or video source, has ever been handled.

**What a newcomer would misread:** A new collaborator might assume the curation method already handles non-English sources or richly visual/diagrammatic material competently, since nothing has ever said otherwise — when in fact the method has only ever been exercised on English text-extractable documents, and its behavior outside that has not been tested even once.

**What should be stated explicitly:** That the curation method's current scope is English, text-extractable sources only, and that non-textual or non-English material is an untested edge case, not a handled one.

---

### 10. Most of "how the Librarian actually behaves" exists only as a pattern across four examples, not as written rules

**How it's actually been operating:** A large number of specific judgment calls — when to escalate to a companion document, how much relationship-confidence nuance to include, when a heuristic is "specific enough" to log, how much of a source to quote versus paraphrase — were made consistently across four cards, but were never extracted into an explicit, standalone statement of practice. They exist only as a pattern a reader could infer by comparing the four cards closely.

**What a newcomer would misread:** A new collaborator reading only the scaling-librarian skill definition and FOUNDATION.md would get the stated philosophy (preserve sources, grade confidence, stay neutral) but would have to reverse-engineer a large amount of actual working practice by close-reading four examples — a slow, lossy way to transmit a working method compared to having it written down.

**What should be stated explicitly:** The specific, recurring judgment calls observed across the four cards so far (several of which are already named in ARCHITECTURAL_REVIEW.md item 1) should be extracted into an explicit practice guide, rather than left as something only visible by comparing outputs.

---

## Why this list matters more than it might seem

Individually, several of these look like minor process footnotes. Together, they point at one thing: a substantial amount of what "Scaling Intelligence" currently *is* lives in this conversation's working memory and in the tacit consistency of one curator's judgment calls — not in the files that would actually survive a move to a new account. Moving to a new account is precisely the event that turns every item on this list from a harmless implicit default into a real risk of misunderstanding. This document exists to close that gap before that move happens, not after.
